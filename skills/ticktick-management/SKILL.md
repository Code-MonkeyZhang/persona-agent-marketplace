# 滴答清单管理规范 (TickTick Management SOP)

## 1. 任务创建原则 (Task Creation)

### 1.1 默认位置
- **规则**：所有新任务**默认创建在"收集箱" (Inbox)**。
- **例外**：只有当用户**明确指定**要把任务放入某个特定清单（Project）时（例如"把这个加到'工作'列表里"），才使用特定的 `project_id`。
- **操作**：调用 `create_tasks` 时，`project_id` 参数默认使用 `"inbox"`。不要自作主张新建 Project。

### 1.2 时间与日期格式
- **规则**：必须使用**带有时区信息的 ISO 8601 格式**。
- **用户所在时区**：默认为 `Asia/Shanghai` (UTC+8)。
- **格式示例**：
  - **正确**: `2025-12-16T09:00:00+08:00` (明确指定了 +08:00)
  - **正确**: `2025-12-16T09:00:00+0000` (UTC 时间)
  - **错误**: `2025-12-16 09:00:00` (无时区，容易导致时间偏移)
- **全天任务**：如果用户未指定具体时刻（如"明天提醒我"），则不设置 `due_date` 的具体时间部分，或者根据 API 要求处理为全天任务。

### 1.3 批量操作
- **规则**：当用户一次性列出多个事项时，**必须使用批量创建/更新**功能（传入 List 数组），严禁循环调用单个 API，以提高效率并减少 API 请求次数。

## 2. 任务查询与汇报
- **范围控制**：在进行"每日汇报"或查询任务时，优先关注 `today` (今日) 和 `overdue` (逾期) 的任务。
- **展示**：向用户展示任务时，清晰列出任务标题、截止时间（转换为用户本地易读格式，如 "14:00"）和优先级。

## 3. 错误处理
- **项目不存在**：如果用户指定的项目名称不存在，**不要自动创建**。应先询问用户："找不到该清单，是否创建新清单，还是放入收集箱？"
- **时间模糊**：如果用户说"晚上提醒我"，默认理解为 **19:00** 或 **20:00**，并在回复中告知用户已设定的具体时间。
